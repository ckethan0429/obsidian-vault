1. 탐색 문제

언제 백트래킹할지

어떤 노드는 병렬 가능하고, 어떤 노드는 반드시 선행 결과를 기다려야 합니다.

3. 비동기 통신 문제

“비동기 노드 통신”도 더 확장하면 그냥 agent-to-agent 대화가 아닙니다.

partial failure를 어떻게 흡수할지

입니다.

4. 스케줄링 문제

slow/fast 작업 분리는 굉장히 큰 축입니다.

앞으로 하네스는 거의 반드시 이렇게 갈 겁니다.

fast lane: 라우팅, 분류, 규칙적 수정, 포맷 정리, 간단한 툴 호출

slow lane: 장기 추론, 설계 대안 비교, 리팩터링, 실험 계획, 원인 분석

priority scheduling

preemption

backpressure

batching

work stealing

deadline-aware execution

입니다.

“어떤 작업을 어떤 SLA와 비용 구조로 실행할까?” 가 하네스의 중심이 됩니다.

같은 중간표현(Intermediate Representation) 으로 변환해야 합니다.

즉 자연어 → 실행이 아니라
자연어 → 구조화된 작업표현 → 실행 → 검증 → 병합
이 되어야 합니다.

이 프레임으로 보면 하네스 엔지니어링은 꽤 많이 컴파일러 설계와 닮습니다.

파싱

정규화

최적화

lowering

codegen

verification

지금 많은 하네스가 이 IR 층 없이 바로 모델에 때려 넣기 때문에 불안정한 겁니다.

8. 장애허용성 문제

에이전트 시스템은 필연적으로 실패합니다.

그래서 하네스는 앞으로 다음을 가져야 합니다.

retry semantics

idempotency

compensating actions

lease / lock

checkpoint/restart

partial rollback

dead-letter queue

human escalation path

즉 “실패하지 않게 만드는 것”이 아니라
“실패를 예상하고 구조적으로 회복하게 만드는 것” 이 핵심입니다.

이건 완전히 분산시스템/워크플로 엔진의 사고방식입니다.

9. 경제성 문제

하네스는 곧 추론 자원 배분기가 됩니다.

예를 들면:

어떤 단계는 작은 모델로 충분한가

어떤 지점에서 큰 모델을 호출할 가치가 있는가

병렬 실행이 latency를 줄이는 대신 비용을 얼마나 늘리는가

speculative branches 중 몇 개만 유지할 것인가

검증 비용이 생성 비용보다 더 큰가

즉 앞으로 하네스는 기술 문제가 아니라
컴퓨트 자본 배분 문제가 됩니다.

여기서 bandit, adaptive routing, cost-aware planning, marginal utility per token 같은 개념이 들어옵니다.

10. 자기개선 문제

여기까지 오면 마지막은 당연히 meta-harness 입니다.

즉 하네스가 작업만 수행하는 것이 아니라,

실패 패턴을 수집하고

어떤 노드가 병목인지 찾고

어떤 handoff 포맷이 더 좋은지 비교하고

어떤 verifier가 false positive/negative를 내는지 보고

자기 자신의 구조를 업데이트

하게 됩니다.
