# Virton 릴리즈 전략 정리

> B2B 패키지 솔루션 기준, 7명 규모 / 10개 엔터프라이즈 고객 컨텍스트

---

## 1. 핵심 개념: 개발 사이클 ≠ 릴리즈 사이클

현재 구축된 건 **Continuous Delivery 파이프라인** (main 머지 시 자동 빌드/패키징).  
하지만 B2B 패키지 솔루션에서 "배포(release)"는 다른 의미.

```text
[개발 배포 (Internal)]  →  [릴리즈 후보 (RC)]  →  [고객 릴리즈 (GA)]
 main 머지 시마다         10-15일 주기 선별        분기/반기 주기
 자동 빌드                 QA 검증                  영업/CS 싱크
```

영업이사가 묻는 건 가장 오른쪽, **GA(General Availability) 주기**.

---

## 2. B2B 솔루션 업계 표준: Semantic Versioning 3-트랙

Proxmox, VMware, Nutanix 등 인프라 솔루션이 공통적으로 쓰는 구조.

| 구분 | 버전 형식 | 주기 | 내용 |
|------|----------|------|------|
| **Major** | X.0.0 | 반기~연 1회 | 아키텍처 변경, 호환성 깨지는 변경 |
| **Minor** | X.Y.0 | 분기 1회 | 신규 기능 추가 |
| **Patch** | X.Y.Z | 월 1회 또는 수시 | 버그 픽스, 보안 패치 |
| **Hotfix** | X.Y.Z | 긴급 시 | 치명 장애/보안 이슈 |

**Virton 권장 주기:**
- Minor 릴리즈: **분기 1회 (연 4회)** ← 영업이사에게 제시할 공식 주기
- Patch 릴리즈: **월 1회** 또는 이슈 발생 시
- Hotfix: **수시**

## 3. Git 브랜치 전략 [아키]

### 구조

```text
feature/*     →  main (개발 통합, CI/CD 자동 배포 = 내부 테스트 환경)
                   ↓  (분기별 선별)
                 release/v2.3.x  (QA, RC 빌드, 고객 전달용 패키징)
                   ↓
                 tag: v2.3.0 (GA)
                   ↓
                 hotfix/* → release/v2.3.x → main (back-merge)
```

### 원칙

- `main`: 계속 빠르게 굴러감 (dev velocity 유지)
- `release/*`: cherry-pick 또는 merge로 **선별된 커밋만** 반영
- 고객에게 나가는 패키지는 **반드시 release 브랜치 태그에서만 빌드**

---

## 4. 릴리즈 트레인(Release Train) 모델 [고]

분기 단위 기차 시간표. SAFe 및 대형 솔루션 회사 표준.

| 주차 | 활동 |
|------|------|
| Week 1-8 | 기능 개발 (main 머지 자유롭게) |
| Week 9 | **Feature Freeze** — release 브랜치 컷 |
| Week 10 | QA 집중 테스트, RC1 배포 |
| Week 11 | 버그 픽스, RC2 배포, 일부 고객 파일럿 |
| Week 12 | GA 릴리즈, 릴리즈 노트 공개, 영업/CS 교육 |

**기차는 놓치면 다음 분기로 자동 이월.**
→ "이 기능 때문에 릴리즈 미뤄달라" 요청을 구조적으로 차단.

## 5. Feature Freeze 이후 브랜치 운영

### 핵심: main은 절대 멈추지 않는다

Feature Freeze = "release 브랜치에 새 기능 안 넣는다"
≠ "main 개발 중단"

```text
Week 1-8:  main ←── feature/* 자유 머지 (기능 개발)

Week 9:    release/v2.3 브랜치를 main에서 컷! ✂️
           │
           ├─ release/v2.3 : Frozen (버그픽스만)
           │
           └─ main : 계속 굴러감 (v2.4 기능 머지 OK)
```

### 9주차 이후 동시 진행

**release/v2.3 (Frozen):**
- 새 기능 머지 ❌
- QA 발견 버그 픽스만 ⭕
- RC1, RC2 빌드
- 최종 GA 태깅

**main (계속 열림):**
- v2.4 기능 머지 ⭕
- 다음 분기 릴리즈 트레인 준비

### 버그픽스 양방향 반영 (중요)

```text
QA가 버그 발견
   ↓
fix 커밋 → release/v2.3 (RC 빌드에 반영)
   ↓
cherry-pick 또는 back-merge → main (v2.4에도 반영)
```

> GitHub Actions에 **자동 back-merge PR 생성** 워크플로우를 걸어두면 편함.

### Hotfix 흐름

```text
hotfix/v2.2.1 브랜치 컷 (from tag v2.2.0)
   ↓
픽스 후 태깅 → v2.2.1 GA
   ↓
back-merge → release/v2.3 + main (둘 다 반영)
```

## 6. 예외 처리 룰

| 상황 | 원칙 | 예외 처리 |
|------|------|----------|
| 9주차 직전 머지 기능을 급하게 v2.3에 넣고 싶을 때 | "기차 놓쳤으니 v2.4" | CTO/영업 승인 시 cherry-pick + RC 재빌드 |
| feature 브랜치 개발 중 9주차 도달 | 그대로 feature에서 작업 지속 | 완성 시 main 머지 → 자연스럽게 v2.4로 |
| GA된 버전에 치명 버그 | Hotfix 브랜치로 즉시 대응 | back-merge 필수 |

> 예외는 분기에 1건 이하로 제한. 이걸 자주 허용하면 Release Train이 무너짐.

## 7. 고객 유형별 트랙 (선택사항) [중]

엔터프라이즈 고객이 모두 동일 주기를 원하지 않을 수 있음.

- **Stable Track**: 반기 1회 (금융/공공 등 보수적 고객)
- **Current Track**: 분기 1회 (최신 기능 선호 고객)
- **LTS 지정**: 예) v2.0은 24개월 지원 보장

> 현재 10개 고객 규모면 오버엔지니어링. **일단 단일 트랙(분기)으로 시작하고 LTS 개념만 걸어두는 것** 권장.

---

## 8. 영업이사에게 답할 스크립트

> 개발 CI/CD와 고객 릴리즈는 별도 트랙으로 운영합니다.
>
> - **정기 릴리즈**: 분기 1회, 신규 기능 포함 (v2.3, v2.4, …)
> - **패치 릴리즈**: 월 1회, 버그/보안 픽스 (v2.3.1, v2.3.2)
> - **핫픽스**: 긴급 이슈 시 즉시
>
> **영업 관점 활용 포인트:**
> - 분기 릴리즈 2~3주 전 Release Note 초안 공유 → 기존 고객 업셀 자료로 활용
> - 신규 제안 시 "분기별 정기 업데이트 + 24개월 LTS 보장" 문구 삽입 가능
> - 주요 기능은 사전에 베타 고객 1~2곳과 검증 후 GA → 레퍼런스 확보

## 9. 실무 체크리스트 [중]

### 문서화
- [ ] 릴리즈 노트 템플릿 (고객용 / 내부용 분리)
- [ ] 버전 네이밍 규칙 문서화 (SemVer 기준)
- [ ] 업그레이드 가이드 문서 (고객별 전달 프로세스)
- [ ] EOL(End of Life) 정책 문서

### 역할/프로세스
- [ ] 릴리즈 매니저 역할 지정 (CK 또는 시니어 개발자)
- [ ] QA 체크리스트 표준화 (회귀, 호환성, 업그레이드 경로 테스트)
- [ ] 롤백 시나리오 정의 (패키지 솔루션은 특히 중요)

### GitHub Actions 워크플로우 분리
- [ ] `main` push → dev 환경 자동 배포 (현재 존재)
- [ ] `release/*` push → RC 패키지 빌드 + 내부 QA 환경 배포
- [ ] `tag v*` → GA 패키지 빌드 + 고객 배포용 아티팩트 서명
- [ ] Hotfix → release/main 자동 back-merge PR 생성

## 10. 놓치기 쉬운 포인트

### 업그레이드 경로(Upgrade Path) 매트릭스
v2.1 → v2.3으로 한번에 가는지, v2.1 → v2.2 → v2.3 거쳐야 하는지 명시.
Proxmox도 문서화되어 있음. 고객 온사이트 엔지니어 없으면 이게 지옥.

### EOL(End of Life) 정책
"v2.0은 2026년 6월까지 보안 패치 지원" — 영업 계약서에 들어갈 수준으로 공식화.

### 릴리즈 노트를 "영업 자산"으로
단순 체인지로그 ❌ → 고객 benefit 관점 ⭕
→ 영업이 재계약/업셀 타이밍에 그대로 들고 나갈 수 있음.
